# サブ回路

再利用可能な回路テンプレート（サブ回路）を導入し、回路構造の再利用を可能にする。

作成日: 2026-03-27
ステータス: 設計完了（未実装）

## 背景・動機

現行の回路定義は全てのワイヤをフラットに列挙する方式であり、加算器のような汎用的な回路構造を再利用する手段がない。大規模な回路では同一パターンが繰り返され、可読性・保守性が低下する。

サブ回路を導入することで:

- 回路テンプレートを定義し、複数箇所でインスタンス化できる
- 各インスタンスは独立した状態を持ち、共有しない
- サブ回路の中でさらに別のサブ回路を参照可能（循環依存は禁止）

## 設計概要

### 概念モデル

2 つの概念を導入する:

1. **サブ回路定義** (`sub_circuits`): 再利用可能な回路テンプレート。ローカル座標系を持ち、入出力インターフェースを宣言する
2. **モジュールインスタンス** (`modules`): サブ回路定義を親回路に配置したもの。入出力セルの親座標を指定する

### シミュレーション方式: 階層的シミュレーション

サブ回路を座標リマッピングによりフラット化する方式も検討したが、座標空間の確保が困難であることと、遅延モデル（座標の辞書順に依存）の保全が複雑になるため、**階層的シミュレーション**を採用する。

- 各モジュールインスタンスは独自の `Simulator` を保持する
- 親の tick 処理中、モジュールの最初の出力セルに到達した時点でサブ回路を 1 tick 実行する
- 入力値は親の現在状態から読み取り、出力値をサブ回路の計算結果で設定する
- サブ回路の内部状態は tick 間で保持される（順序回路に対応）

### ユーザー提案からの変更点

ユーザーの初期提案に対して、以下の設計上の問題を是正した:

| 項目 | ユーザー提案 | 変更後 | 理由 |
|---|---|---|---|
| 定義構造 | `"sub": [ "sub1": {...} ]` (配列+キー) | `"sub_circuits": { "sub1": {...} }` (オブジェクト) | 元の形式は JSON として不正。オブジェクトマップに変更 |
| インスタンス | `"module"` | `"modules"` | 複数形に統一（JSON 配列） |
| 参照方法 | `"type": "sub"` + `"id": "sub1"` | `"type": "sub1"` | `type` でサブ回路定義名を直接参照。`id` は不要 |
| 定義内 type | `"type": "circuit"` | 削除 | `sub_circuits` 内の定義は全て回路であり冗長 |
| キー命名 | `"sub-input"`, `"sub-output"` | `"sub_input"`, `"sub_output"` | snake_case に統一 |
| 定義スコープ | ネスト定義 | ルートレベルのみ | 初期実装の簡素化。ネスト定義は将来拡張として検討 |
